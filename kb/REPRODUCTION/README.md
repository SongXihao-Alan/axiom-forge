# Axiom Forge 复现任务包

> 目标:**让 2-3 个有基础训练的人**(知道博弈论 + feature attribution + 写过论文)能**独立探索** Axiom Forge,
> **找到**他们自己的 axiom,记录过程,评估质量。
>
> 这不是"测试",是 **AI4SS 的"过程示范"**——我们想验证:工具中立性是否真中立?探索过程是否可追溯?

---

## 1. 谁能用(受众)

| 必须有 | 不必有 |
|---|---|
| 知道 cooperative game theory 基本概念 | 懂 LLM 内部 |
| 知道 feature attribution(SHAP / LIME / IG 至少用过一种) | 懂 Python / pipeline 内部 |
| 读过 Thomson 或同类型方法论文 | 懂 AI4SS 框架 |
| 写过学术论文(IEEE / ACM / Springer 格式) | 懂 KB schema |
| 愿意写反思(不只给结果) | 同意"找到正确 axiom"这种目标 |

**3 类典型受众**:
- **A 类**:合作博弈论方向的研究生(强项:博弈论;弱项:ML attribution)
- **B 类**:ML / XAI 方向的研究生(强项:attribution;弱项:博弈论/哲学)
- **C 类**:经济学/政治哲学方向的研究生(强项:价值观锚;弱项:技术细节)

**所有 3 类**都能用——任务设计**不强求技术细节**,强调**思考 + 记录**。

## 2. 复现任务(3 个,选 1)

### 任务 T1:**独立探索一个新 axiom**

**目标**:在 KB 已有 85 节点基础上,**提出 1 个新 axiom**(不在 KB 中),并完成:
- 形式化(自然语言 + 数学)
- 3 锚分析(经验 / 哲学 / 社群,任选 1-3 个,不需要都满足)
- 反例 1 个(具体场景下该 axiom 失败)
- 跟 KB 已有 axiom 的关系(parent / contradict / generalizes)

**步骤**:
1. `./axiom-forge list` —— 浏览 KB,看已有 axiom
2. `./axiom-forge anchors-by-type community` —— 看 3 锚分布
3. `./axiom-forge value-tree` —— 看价值谱系
4. **找一个"KB 缺什么"** —— 可能是某个域(moral/interest/...) 缺 axiom
5. 用 CLI 验证: `./axiom-forge query <你的 axiom 关键词>`
6. 写新 axiom JSON(参考 `kb/nodes/axioms/AX-SC-001.json` 格式)
7. 提交:1 个 JSON + 1 段反思(200-500 字)

**预期时间**:4-6 小时

### 任务 T2:**找 5 个 SC 违反案例**

**目标**:在 KB 已有 Impossibility 5.1 基础上,**找 5 个新场景**,每个场景构造一个 f, f̂ 使得 SC 违反。

**步骤**:
1. `./axiom-forge show TH-IMP-501` —— 看已有反例(f(X) = βX_1, f̂(X) = 0)
2. **找 5 个新场景**:
   - 场景 1:非线性 f(eg `f(X) = X_1^2`)
   - 场景 2:f̂ 跟 f 是 2 个不同的 ML 模型(eg XGBoost vs 简单线性)
   - 场景 3:f 有 confounding(eg `X_1 → X_2 → Y`,f̂ 只看 X_2)
   - 场景 4:有 mediator(eg `X_1 → M → Y`,f̂ 看 M)
   - 场景 5:f̂ 有 spurious correlation(eg f̂ 把 noise 误认为信号)
3. 每个场景给出:**f / f̂ / 验证 SI > 0 + φ = 0 / 哲学/经验 锚解释**
4. 提交:1 个 markdown, 5 个场景各 200 字

**预期时间**:3-5 小时

### 任务 T3:**为 KB 贡献 5 个新 value_anchor**

**目标**:在 KB 已有 33 个 value_anchor 基础上,**加 5 个新的**,覆盖 6 大类中**没覆盖的子类**。

**步骤**:
1. `./axiom-forge value-tree` —— 看 6 大类
2. **找 5 个缺**:
   - 道德类:可能缺"忠诚" / "感恩" / "勇气"
   - 利益类:可能缺"自由时间" / "数据所有权"
   - 美学类:可能缺"幽默" / "对比"
   - 知识类:可能缺"谦逊"(epistemic humility)
   - 实践类:可能缺"安全性" / "用户主权"
   - 哲学类:可能缺"佛教缘起" / "非洲 Ubuntu" / "儒家仁"
3. 每个 value_anchor 给出:
   - label_zh + label_en
   - value_class + value_subclass
   - cross_cultural_consistency 评估(high/medium/low)
   - anchors_self(至少 1 个)
4. 提交:5 个 JSON 文件

**预期时间**:2-3 小时

## 3. 工具使用(最短路径)

```bash
# 安装
cd axiom-finder
chmod +x axiom-forge

# 5 个最常用命令
./axiom-forge list                          # 全 KB 概览
./axiom-forge value-tree                    # 价值谱系
./axiom-forge search "你的关键词"            # 语义搜索
./axiom-forge show <NODE_ID>                # 节点详情
./axiom-forge relations <NODE_ID>           # 节点关系
./axiom-forge graph <NODE_ID> --depth 2     # 关系图
./axiom-forge anchors <NODE_ID>             # 看节点的所有锚
./axiom-forge anchors-by-type <type>        # 按锚类型列节点
./axiom-forge compressed <NODE_ID> [粒度]   # 多粒度
./axiom-forge stats                         # 统计

# 高级
./axiom-forge help                          # 全部命令
```

**不需要**:
- Python 编程(只是用 CLI)
- M3 API key(CLI 不调用 LLM)
- 网络(本地文件)
- 安装额外包(只用 stdlib `json`, `pathlib`, `re`)

## 4. 评估标准(给任务完成者)

我们**不是**给"对/错"评分——3 锚**没有优劣**。我们评估**过程质量**。

### 4.1 任务 T1 (新 axiom) 评估

| 维度 | 评分点 | 满分 |
|---|---|---|
| **形式化** | 自然语言 + 数学清晰 | /3 |
| **锚选择** | 至少 1 个锚(经验/哲学/社群任选),**解释为什么** | /3 |
| **反例** | 1 个具体反例,说明 axiom 何时失败 | /2 |
| **跟 KB 关系** | 显式说跟哪些节点 parent / contradict | /2 |
| **过程反思** | 200-500 字:为什么提这个 axiom?走了哪些弯路? | 5 (满分) |

**15 分 + 5 反思 = 20**

### 4.2 任务 T2 (5 个反例) 评估

| 维度 | 评分点 | 满分 |
|---|---|---|
| **多样性** | 5 个反例覆盖**不同场景**(非线性/模型差异/confounding/mediator/spurious) | /5 |
| **严格性** | 每个反例都验证 SI > 0 + φ = 0 | /5 |
| **锚解释** | 每个反例都说明哲学/经验 锚 | /3 |
| **跟主线的关联** | 说明这些反例如何**强化** Impossibility 5.1 | /2 |
| **过程反思** | 200-500 字:找反例时有什么困难? | 5 |

**15 分 + 5 反思 = 20**

### 4.3 任务 T3 (5 个 value_anchor) 评估

| 维度 | 评分点 | 满分 |
|---|---|---|
| **多样性** | 5 个覆盖 6 大类中**至少 3 个不同类** | /3 |
| **跨文化一致性评估** | 评估有理由(不能瞎填 high/medium/low) | /3 |
| **anchors_self 真实** | 不是空填,有具体 evidence | /2 |
| **可整合性** | 跟 KB 已有 33 个 value_anchor 显式关联 | /2 |
| **过程反思** | 200-500 字:为什么选这些? | 5 |

**10 分 + 5 反思 = 15**

## 5. 给任务完成者的"工具中立性"反馈表

复现完成后,**所有任务完成者**需要填:

```markdown
## 工具中立性反馈

1. **你提出的 axiom 用了哪几个锚?**
   - [ ] 经验锚(具体: ___)
   - [ ] 哲学锚(具体: ___)
   - [ ] 社群锚(具体: ___)

2. **CLI 是否预设了什么优先?**
   - 比如: "anchors-by-type 是否有隐藏的优先级?"
   - 比如: "value-tree 6 大类的顺序是否暗示优先级?"

3. **有没有觉得"被引导向某个结论"?**
   - 比如: "AX-SC-001 节点已经存在, 是否让我倾向 '沿着 SC 走'?"

4. **你提的 axiom 跟 KB 已有 axiom 的关系是?**
   - [ ] Parent (扩展)
   - [ ] Sibling (并列)
   - [ ] Contradict (矛盾)
   - [ ] Independent (独立)

5. **如果换一个人, 提的 axiom 会不会不同?**
   - [ ] 会 (因为锚不同)
   - [ ] 不会 (因为问题只有 1 个解)
```

**这 5 个问题的答案**会帮我们评估**工具中立性**。

## 8. 关键提醒(给任务完成者)

1. **不要追求"对"**——3 锚没有优劣,你的判断跟别人不一样是**正常**的
2. **记录过程**——卡在哪里、改了什么、为什么改,比结果重要
3. **不要假设"深 = 好"**——找到 1 个有效的锚就够,不用 3 个都满足
4. **CLI 是工具不是老师**——它给你数据,不给判断
5. **可以不同意 KB**——你觉得某个 axiom 错了,直接说 + 写新 axiom
